## Assistant abuse-hardening (anti-Anthropic-ban)
Two complementary caps now bound token spend:
- **Per-request input size (PR #169).** `src/lib/agent/limits.ts` — `MAX_MESSAGE_CHARS=1000`.
  `sendToAssistant` **rejects** an oversized latest message *before any model call* (zero tokens);
  `runAgentTurn` defensively `clampContent`s every message. Client: input `maxLength` + voice capture
  hard-stopped at `MAX_VOICE_MS=30s`. (Web Speech sends transcribed *text*, not audio — so the char
  cap covers the "too much audio" vector too.)
- **Per-user volume / rate limit (PR #171).** Migration `20260606T1000_agent_rate.sql` (**applied to
  prod**): `agent_rate` fixed-window counters + **`bump_agent_rate(p_hour_limit, p_day_limit)`** — a
  `SECURITY DEFINER` fn scoped to `auth.uid()` that atomically upsert-increments the hourly + daily
  windows and returns within-limit; self-cleans windows >2d. `sendToAssistant` gates before any model
  call (over-limit → friendly message, zero tokens), **fail-open**. `RATE_PER_HOUR=30`/`RATE_PER_DAY=150`.
  **Security gotcha (logged in [[security-audits]]):** Supabase default privileges auto-grant EXECUTE to
  `anon` on new public fns, so `REVOKE … FROM PUBLIC` is insufficient — must also `REVOKE … FROM anon`
  (advisor caught it; ACL now matches the hardened `bump_app_usage`; both in the migration file).
