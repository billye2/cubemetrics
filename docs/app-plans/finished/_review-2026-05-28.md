# App-plan review — 2026-05-28

A walk-through of every app against its plan in this folder, plus cross-cutting improvements the plans don't yet capture. Written from the code, not from memory.

## TL;DR for the senior dev

1. **The plans are systematically out of date.** Four of the five factory templates (Checklist, Logbook, Goal, Finance) describe a thinner shell than what actually ships. Most of the P1 items in `_checklist-template.md`, `_logbook-template.md`, `_goal-template.md`, and `_finance-template.md` have already landed. Recommend refreshing each plan's *Current state* section and pruning the done items before any new template work starts.
2. **`_tracker-template.md` P1 just shipped** (this session — PR #10): aggregation modes, stats strip, 7-day chart, streak. P2/P3 are still untouched.
3. **The README still calls Focus/TimeTracker/Countdown the only reference apps.** Goal and Finance factories are arguably at-bar now too — both have a hero, a stats strip, a visualization, and adaptive formatting. Worth bumping their status in the README.
4. **Two custom apps have a single highest-impact gap each** that the plans correctly identify: Calendar (no month grid + ignores `end_*`/`recurrence` columns), Pomodoro (no work/break cycle automation). These are the cleanest "do this next" opportunities.
5. **A handful of custom apps still have no edit action** (Journal, Notes, Todo, Calendar, Reading content, Habits name, Countdown). The plans flag each individually; consider a sweep — adding `update*Action` and an inline-edit pattern shared across them is one PR's worth of work.

## Plan-freshness audit

What the plans say vs. what the code does. None of these are bugs in the *apps* — they're staleness in the *docs*.

### `_checklist-template.md`
Plan says: "single flat list: add a title, check it off, delete, collapse completed." Code already has:
- Optional `note` column wired into form + row, with auto-detection of `https?://` URLs (renders as link).
- Progress header ("N of M done", percent bar, emerald at 100%).
- Sort toggle ("To do first" / "Newest" / "A–Z") shown when total > 2.
- Collapsible done section.

So P1 #1–#3 in the plan are already shipped, except **reorder/manual sort** (still missing) and `quantity`/`section`/`due_date` columns (still missing).

### `_logbook-template.md`
Plan says edit / search / month grouping / backdate are P1, prompts are P2. Code already has all of them:
- `logbookUpdateAction` + an inline edit mode on every entry card.
- Search filter (shows when entries > 3).
- Month grouping with per-month counts.
- Backdate via an `entry_date` input on the add form (anchored to local noon to preserve the calendar day).
- Per-log-type prompts driven by an internal `PROMPTS` map (gratitude, meeting, standup, brainstorm, decision, workout, learning, weekly, feedbacklog, oneonone, retro, recipe). Driving these from `config` per the plan would still be cleaner.

Still missing: tags, markdown rendering, and the four "structured" graduations (workout, recipes, decisionmatrix, weeklyreview).

### `_goal-template.md`
Plan says deadlines, progress history, increment buttons, notes are P1. Code already has:
- `due_date` with sort-by-deadline-ascending, "overdue" red border + label, "due soon" amber label.
- `description` text field shown on each card.
- `+1` quick-increment button + "Set value" inline editor.
- Stats strip (Active / Avg progress / Due ≤7d).

Still missing: **progress history** (the plan's `goal_progress` table — progress updates still overwrite the latest value), key results / sub-tasks, currency/pace math for savings/debt, completion archive view.

### `_finance-template.md`
Plan says recurrence, per-type totals, category breakdown, `Intl` currency are P1/P2. Code already has all four:
- `frequency` column wired up (`monthly`/`weekly`/`yearly`/`quarterly`) with `monthlyFactor` normalization in `factoryLib.ts`.
- Subscription hero shows monthly + annual + active count; bill/invoice hero shows outstanding + overdue count.
- Top-5 category breakdown with horizontal bars (`currency()` formatter, monthly-normalized for subscriptions).
- `currency()` / `currencyCompact()` via `Intl.NumberFormat`.
- Due-date urgency: overdue red border + red label, due-soon amber label, sort by date.

Still missing: **`paid_date` history** (paid items still just hide from the unpaid list — no trend across months), budget app's planned-vs-actual, expense-app integration.

### `_tracker-template.md`
P1 just shipped. Plan should be updated to mark P1 done.

## Per-app additions (beyond what the plans propose)

These are improvements the existing plans don't cover. I've kept them small and specific.

### Todo (`todo`)
- **Empty state should be opinionated**, not passive. After the user clears their list, the empty state could read *"All clear — nice."* and show "completed today" count from `completed_at` instead of the generic prompt.
- **Keyboard shortcuts on desktop** — pressing 1/2/3 while the title input is focused cycles priority Normal/`!`/`!!` (the priority pills are already there; this is just wiring). Phone-first stays the default.
- **Long-press "✓" to bulk-clear all completed.** Avoids a discrete "Clear completed" button while staying mobile-friendly.

### Journal (`journal`)
- **Live word count on the editor** + per-entry word count on cards. Cheap, motivating, and "Words written this month" (currently a P3 stat) lights up for free once it's per-entry.
- **`/app/journal/new` is its own route** — fine for a longer form, but a returning-user flow benefits from inline edit on `/app/journal` once the edit action lands. Consider folding new+edit into a shared `EntryForm` component to avoid drift.
- **Mood emoji set isn't ordinal.** Today the 7 emoji `😊😌😐😔😤😴🤔` are unordered (`sleep`/`thinking` aren't on the happiness axis). To make the planned "mood-over-time chart" honest, either pick an ordinal subset (5-point Likert) or store both an ordinal axis and a non-ordinal "tag" emoji. Worth flagging *before* the chart ships.

### Notes (`notes`)
- **URL paste detection.** If the body starts with `https?://`, suggest setting the title to the page's `<title>` (cheap fetch through a server action). Turns the app into a usable read-later.
- **`updated_at` is read but never written** — a bug today (no edit action means it always equals `created_at`). Edit action lands first, then this is informative.
- **Tags column already in schema** — surface it before adding folders. Folders is the plan's P3; tags solve most of the grouping need with the table you already have.

### Calendar (`calendar`)
- **Read-only cross-app overlay (no edit, no schema).** Surface `countdowns.target_date` and `finance_items.due_date` as muted day-dots on whichever view ships first (agenda or month). The plan calls this P3; it's actually one query each and gives the calendar real density immediately.
- **The agenda Past section uses a separate query.** Once edit/recurrence land, recurrence-expanded occurrences won't appear in the Past list because they don't have rows. Worth deciding the model now: expand at read-time across the whole 60-day window (or 120-day) and partition into upcoming/past client-side.

### Pomodoro (`pomodoro`)
- **The plan's headline is right: ship the work/break cycle.** No new addition needed there.
- **Move active state to `localStorage` to match Focus.** Two things make this worth doing: (a) the cycle is multi-phase, so the source of truth is the *cycle*, not a single row; (b) the current server-row design loses information when the user closes the tab mid-break — the next session starts a fresh "work" phase instead of resuming. Only completed work sessions need a row.
- **Notification permission ask should be lazy** (on first start), not on page load. Common dark pattern to avoid.

### Habits (`habits`)
- **Backfill on a missed day.** Most realistic habit-app failure: you forget to check in on a busy day. Long-press a habit tile to backdate a check-in (date picker, capped to last 14 days) — that one affordance protects the streak that motivates the user.
- **Move check-ins under `created_at` not `checkin_date`?** Schema has both. `checkin_date` is the canonical day; `created_at` is the click timestamp. Today's "streak" reads `checkin_date`, which is correct — just double-check nobody else reads `created_at`.
- **Soft-delete with no archive UX is a bug-magnet.** A user who "deletes" a habit and then creates one with the same name silently has an orphan `active=false` row. Either restore-on-recreate or surface an Archived list (the plan's P2).

### Expenses (`expenses`)
- **Receipt photo (P3) is more valuable than it looks.** People log expenses *to* keep receipts; defer it if you must, but the recurring-expenses idea above it is something most users won't notice.
- **The fixed 8-category list is duplicated in `page.tsx` and `actions.ts`.** Move it to `src/lib/modern/expenseCategories.ts` until per-user categories ship — one source of truth and one place to add a color map.
- **Hero card shows month and week totals as equally weighted.** Week-of-month varies wildly (1/4 of a month, 1/2, …). Prefer "This month / Daily avg this month" or "Today / This month" — both more interpretable than "This week" in isolation.

### Reading (`reading`)
- **Rating is hidden until completed, but the action allows it at any status.** Either show stars on `reading` cards too (you might want to set an interim rating for re-reads / DNF) or make the action match the UI. Today's mismatch is confusing.
- **Status dropdown is the *only* way to move books.** A "Start reading" / "Mark finished" CTA on the dominant card type is faster than opening a dropdown. Especially on phone.
- **Plan suggests a yearly goal ring.** Suggest storing it in `localStorage` first as the plan says — but expose a tiny "set goal" link inline in the stats strip rather than a separate settings screen.

### Feedback (`feedback`)
- **`github_issue_number` is stored but never displayed.** Showing `#42` next to the "View issue ↗" link in the Mine tab makes the loop feel concrete.
- **"Shipped" state isn't just a status widening.** It's a poll loop against the linked issue's `state`/`merged` — fold that into the existing approve action via a daily scheduled routine (the `@claude` Action already exists), rather than re-inventing a separate poller.
- **The Board tab is currently a shared read of `user_feedback`.** Worth re-checking RLS: the public-read policy was dropped in migration 015, so the Board now only shows the *submitter's* rows under RLS. Either restore a narrower Board policy (e.g. share only `approved` items) or remove the tab.

### Focus, Time Tracker, Countdown (reference apps)
The plans correctly mark these as complete. From a polish pass:
- **Focus**: `localStorage` payload should include a schema version. A future change to the shape will silently break in-flight sessions for returning users.
- **Time Tracker**: stable category colors are derived from order-of-first-appearance — that means deleting the oldest entry can re-assign colors. Persist the chosen color on first use (or compute from a hash of the category name) for stability.
- **Countdown**: the 1-second `setInterval` runs forever even when no card is "imminent". A small optimization: tick every 60s when no card has `granularity === "imminent"`, drop to 1s when one becomes imminent.

## Cross-cutting improvements (apply broadly, not in any plan)

These are the kinds of things that lift every app at once. Pick the ones that match what users actually complain about.

1. **Reduced-motion respect.** All the progress bars, bar charts, and modal/expand transitions animate. Wrap the relevant `transition-*` classes in `motion-safe:` so the OS preference is honored.
2. **Per-page `<title>`.** Every page renders as "XP Boost" in the browser tab. Adding a `metadata` export per route (`title: \`${appName} · XP Boost\``) is one line and makes multi-tab use bearable.
3. **`confirm()` for destructive ops is jarring on phones.** Pattern this session uses everywhere. A 5-second undo toast is more phone-idiomatic and removes the modal interrupt. Consider a tiny `<UndoToast>` primitive in `components/modern/`.
4. **Server-action error surfaces.** Today `revalidatePath()` happens whether the write succeeded or not. RLS denials or Postgres errors are silent to the user. A thin wrapper that catches and surfaces an inline error would catch most "I tapped Save but nothing happened" reports.
5. **Pagination / load-more on history lists.** Tracker now loads 60 days but caps display at 30 rows; most logbooks fetch 100; recent-sessions lists cap at 20. None of them have a "Load more" — the data is just gone from view. A shared `<LoadMore>` component would fix every app at once.
6. **A shared chart primitive.** Three apps (Focus, Time Tracker, Countdown reference impls) plus the new Tracker chart all hand-roll the 7-day bar pattern. Extracting `<DailyBarChart buckets={...} accent="cyan" />` would reduce drift; the *next* app (Pomodoro 7-day, Calendar density, Habits heatmap) would compose it.
7. **`createServerSupabase()` is called twice on most `page.tsx` files** (once for `auth.getUser`, again implicitly inside the queries). Already efficient as long as the cookie store is the same, but worth confirming the singleton boundary.
8. **No catalog entry has a `tags` field**, so the global search at `/` searches name + description + category only. Once apps gain semantic tags (e.g. "money", "writing", "habits"), search gets much better.
9. **`docs/app-plans/README.md` mentions "60+ apps" in some places but the catalog is currently ~50.** Reconcile the numbers before the next user-facing claim.
10. **The plan files use a mix of effort markers** ("S/M/L" verdict, "P1/P2/P3" priority, plus prose). Pick one and stick to it for the next round of plans.
