### Tracker P1 ‚Äî **At spec**

### Goal P1 ‚Äî **Slightly below spec**

**Code:** `GoalView.tsx`. Shipped: deadlines (`due_date` with sort + color), `+1`/Set-value increments, `description` field, plus a stats strip (Active / Avg progress / Due ‚â§7d) the plan didn't ask for.

The gap:
- **No `goal_progress` table.** `goalUpdateProgressAction` overwrites `current_value`. The plan calls this out as the *headline* P1 item ("Today updating progress overwrites the number and loses history"). Without a history row per update, the planned trend / pace math is unbuildable.

### Pomodoro ‚Äî **Mostly at spec, dipped into P3**
**Plan:** `pomodoro.md` P1 ‚Äî automatic cycle + completion sound + Notification. P2 ‚Äî configurable durations, 7-day chart, daily goal. P3 ‚Äî ambient sound, auto-start toggle, task linkage.
**Code:** `src/app/app/pomodoro/{page.tsx, actions.ts, PomodoroView.tsx}`.

What's there: full work/break cycle with `CycleDots`, long-break-every-N rule (`perLong` setting), chime on phase change, `Notification` ask + emit (lazy on first start), 7-day chart, settings panel for short/long minutes + `perLong` + autoStart + sound. The `autoStart` toggle is a P3 item ‚Äî pulled forward.

The gap from P2:
- **No daily-pomodoro-goal UI.** The plan asks for a target (e.g. 8/day) and a stats-strip progress against it. The hero currently shows today's count alone.

Architectural note worth flagging:
- The plan recommended *moving the active cycle to `localStorage` to match Focus*. Current code is **hybrid**: active *work* session is a server-side `pomodoro_sessions` row, *break* state is a `localStorage` blob. This works, but it means closing the tab mid-break and reopening behaves differently from closing mid-work. If you ever want shared-session-across-tabs semantics, the hybrid is fine; if you'd rather match Focus exactly, consolidate the break state into the same server row (e.g. a `kind` column the plan hints at) or move work fully into `localStorage` and only write *completed* work sessions server-side.

Not yet started: ambient sound (P3), task linkage / labels-per-cycle (P3).

### XP layer ‚Äî **Ahead of its own status notes**

**Spec:** `_xp-layer-spec.md` (Phases 1‚Äì4) + `_xp-quests-spec.md` (Phase 3) + `_xp-timezone-spec.md` (Phase 4 timezone).
**Code:** `src/lib/xp/{rules,levels,stats,achievements,compute,quests}.ts` + `src/app/app/xp/page.tsx` + header strip in `src/app/page.tsx` + migration 020.

**What's better than the current-state memory says:**
The memory line says *"Phase 3 = daily quests (table exists, pool not built)"*. That's stale ‚Äî `quests.ts` is fully built: 11 quest defs, deterministic FNV-1a hash pick, `metricsFromActivity`, `questStatuses`, `questPointsForDay`, auto-claim wired into `ensureXp`, idempotent inserts via PK. The dashboard renders the today-quests section. Recommend updating the memory.

**What's at spec:** rules registry shape, level curve `xpToReach(L)=50(L-1)¬≤`, level titles, achievements ledger (9 keys), `xp_daily` upsert design, `pickDailyQuests` determinism, `questPointsForDay` derivation from claimed rows (never punish), GLOBAL_DAILY_CAP (the spec said ~250, code uses 300 ‚Äî close enough, comment-only).

**What's still missing:**

1. **Phase 4 timezone ‚Äî not shipped.** The `_xp-timezone-spec.md` design is detailed and ready:
   - `src/lib/xp/tz.ts` is absent.
   - `compute.ts` derives `dayKey()` from `now.getFullYear/Month/Date()` ‚Äî that's the server zone, i.e. UTC on Vercel.
   - `profiles.timezone` column exists in migration 020 but no read/write path uses it.
   - No `TimezoneSync` client component anywhere.
   - **Concrete impact:** any user not on UTC will mis-bucket late-night/early-morning activity. A todo completed at 6pm PST is currently counted as the *next* UTC day ‚Äî inflating one day, breaking the streak, and mis-scoring the quest. Per the existing test files this isn't covered, so users in the affected zones will see drift before any test catches it.
2. **`early_bird` / `night_owl` achievements** are listed in the spec (`_xp-layer-spec.md` ¬ß7) but not present in `achievements.ts`. They need a `firstActionHour` / `lastActionHour` field added to `CumulativeStats`, derived from min/max `created_at` per local day (which then *also* depends on the timezone fix ‚Äî they'd be wrong by hours otherwise).
3. **First-action-in-an-app bonus.** Spec ¬ß3 lists `+5` *first action in an app today* as a row in the economy. Code implements a `BREADTH_BONUS` of 5 per active source instead ‚Äî the math comes out similar (an app's first action triggers the breadth bonus the moment that source becomes "active"), but the semantics differ: spec says "every newly-touched-today app earns +5 once"; code says "for every distinct active source today, +5". Worth noting these end up equivalent in practice since active = touched. Comment in code, not a behavior bug.

**Improvement suggestions not in any spec:**

- The `ensureXp` write-path filters days with `points > 0 OR isToday`. Past zero-XP days don't get a row, which means a later achievement evaluator that wants "100 active days" needs to be careful ‚Äî and `committed` (the 100-active-days achievement) is defined on `activeDays`. Good news: `activeDays` is the count of rows in `xp_daily` after upserts, so this works *as long as today's points-zero row is kept*. The code looks correct here; just worth a one-line comment explaining the invariant.
- The 30-day daily series is computed inside `compute.ts` and returned in the summary. That's correct but means the page can't ask for a longer window without changing the function signature. Consider returning all rows and slicing per surface.
- `xp_daily.computed_at` is updated on every recompute ‚Äî useful for cache debugging, not surfaced anywhere. A "last recomputed" line in the dashboard footer would help cheaply verify recompute-on-visit is firing.
- `achievements.ts` enforces a static catalog, but newly-unlocked badges show "new!" only for the last 24h ‚Äî that read of `unlocked_at` is in compute.ts (`newlyUnlocked`). Worth a unit test pinning the 24h window so a future timezone change doesn't silently widen it.

## Cross-cutting nits and follow-ons

These apply across the round-2 surface, not to a single app:

1. **No `update*Action` for: Calendar, Notes, Todo, Reading, Habits name, Countdown.** First review flagged the pattern; Workout/Flashcards landed without one too (sets/cards are add+delete only, which is fine for sets/cards but means a mistyped exercise name lives forever as a different exercise). A shared `<InlineEdit>` primitive in `components/modern/` would close all of these with one PR.
2. **The breadth bonus / first-action semantics merge** (XP ¬ß3) is worth a single source-of-truth comment in `rules.ts` so the spec and the code don't diverge silently as the economy gets tuned.
3. **Pomodoro and Workout charts are hand-rolled** ‚Äî same 7-day-bar pattern as Focus / Time Tracker / Countdown. First review proposed a shared `<DailyBarChart>` primitive. Now the case is even stronger.
4. **`profiles.timezone` is in the DB but unused.** Two specs depend on it (`_xp-timezone-spec.md`, plus implicit per-app `*_date` write-time correctness). Decide whether to ship `TimezoneSync` as a small standalone PR before any further XP work ‚Äî it unblocks the rest.
5. **`docs/database.md` still flags 014_drop_classic.sql as "not yet applied"** ‚Äî verify against the live database; the production state may have changed since the doc note was written.
